| D-SA-rules | Phase 2 / Static Analysis Rules | `static_analyzer.py` 首版必须实现以下规则，且 `issue_id` 前缀按问题维度分配：<br>1. `type_compatibility`（C-R, warning）：若 `reward.term_expr` 与任一 hard constraint `indicator_predicate` 共享状态变量 token，且该 reward term `weight > 0`，则报告可能的 boundary-seeking 激励。理论依据：CRE_v4 第 8 页 §2.3 规则 (1) `Type compatibility`；本项目以表达式 / 谓词字符串的变量 token overlap 作为可实现的静态近似。<br>2. `domain_boundary`（C-R, warning）：若 reward term `weight > 0` 且 `clip_bounds.min >= 0`，同时与任一 hard constraint 共享变量 token，则报告 reward 支撑可能落在约束边界方向。理论依据：CRE_v4 第 8 页 §2.3 规则 (2) `Domain boundary`；本项目以非负支撑 + 变量重叠作为符号化近似。<br>3. `coverage_prebound`（E-C, warning）：对每个 hard constraint，检查 `E_tr.scene_families` 与顶层 `shift_operators.description` 是否存在可覆盖该约束 critical region 的关键词匹配；若无匹配则报告训练分布覆盖不足。理论依据：CRE_v4 第 8 页 §2.3 规则 (3) `Coverage pre-bound`，其要求每个 hard constraint 至少有一个 scene family 支撑。<br>4. `deployment_shift_coverage`（E-R, warning）：若 `E_dep.deployment_envs[*].applied_shift_operators` 存在非空项，而 `E_tr.shift_operators == []`，则报告训练分布未显式覆盖部署 shift。理论依据：CRE 的 `E_tr / E_dep` 双环境表示要求训练环境对部署偏移给出显式覆盖声明；缺失 shift coverage 是典型 environment-reward 风险。<br>5. `soft_constraint_penalty_alignment`（C-R, info）：对每个 soft constraint，若其 `penalty_weight > 0`，且存在共享变量 token 的 reward term 具有 `weight > 0`，则报告 reward 激励与 soft constraint 惩罚可能相互牵制。理论依据：CRE Class I 关注 reward 与 constraint 的方向性不一致；由于 Phase 2 不解析表达式语义，该规则仅作为低置信度 `info`。<br>统一约束：Phase 2 首版的变量共享与关键词覆盖判断统一采用 token heuristic，不做 AST / SMT / 谓词求解；因此上述规则默认输出 `warning/info`，保留 `error` 等级给未来可证明的结构性矛盾 | CRE_v4 第 8 页 §2.3 给出静态 pre-check 的三条理论约束；本项目因跳过 LLM-α 与正式符号解析，只能以字符串 token / 关键词匹配实现其可操作近似，因此需先冻结规则边界、证据来源与默认严重等级，避免后续实现随意扩大解释范围 | FINAL |
