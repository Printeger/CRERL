Figure C — 以强化学习训练主干为中心的 CRE 介入与迭代流程

这张图把 RL 训练流程 作为唯一主干，竖向按时间顺序排列；左右分支只表达 CRE 在哪里读取日志、做诊断、提出修复、以及在什么时间尺度回写 R / C / E / 配置。 结论是：CRE 是混合介入方式——日志采集发生在每 step / 每 episode，诊断与结构性修复通常发生在每 K 次 rollout / M 次更新 / 一个 checkpoint gate，而不是每个 step 都改规范。
不是每个 episode 都改 R/C/E 训练中的结构性修改通常在阶段性 gate 触发后进行；否则会把 rollout 分布、buffer 统计和优化过程搅乱。
每次 Collect rollouts 都会记录日志 但默认不会在每个 rollout 结束后立刻修改规范。先累计证据，再决定是否进入 CRE repair loop。
典型策略：混合介入 per-step / per-episode 做观测与落盘；per-gate 做 CRE 评分与修复；accepted repair 在下一个 rollout window 前生效。
主干：RL training pipeline 每 step：runtime logging / raw signals 每 episode：episode summary / aggregates 每 gate：CRE diagnose / score / repair 写回点：修改 R / C / E / config 并在后续训练窗口生效
左侧：训练期日志与证据累积
中间主干：强化学习训练流程
右侧：CRE 介入、诊断、修复与回写
一次性

训练前解析与预检

把任务设计物转成初始规格 S₀ = (R, C, E, Π)，并登记将来允许被 repair 的字段。

0
定义初始训练规格与训练计划
设定 reward graph、constraint predicates、training env distribution、deployment eval set、PPO/SAC 等训练配置、gate 周期 K / M。
M1 / precheck

CRE 可介入字段先注册

  • R：term weights、penalty λ、clip bounds、shaping on/off
  • C：阈值、hard/soft、tolerance ζ、curriculum schedule
  • E：scene mix、obstacle density、spawn rules、noise/domain randomization range
启动

初始化日志面板与缓冲

分出 step-level、episode-level、rollout-window-level 三类日志通道，确保后续 CRE 可以读取。

1
实例化环境、策略、优化器、logger、checkpoint manager
构建当前版本的训练环境 Etr、reward/constraint evaluators、policy πθ、critic、replay/buffer、日志器与评估器。
回写入口

这里是“下轮训练窗口”的装载点

如果上一轮 CRE 已接受 repair，这里会载入最新的 S′、更新的 env sampler、reward DAG、constraint config 与 utility oracle config。

每 step

runtime raw logging

  • state / action / reward terms rₖ
  • distance-to-obstacle / clearance / near-boundary 信号
  • constraint indicator Oⱼ(s,a)、violation timestamp
  • env params：障碍密度、噪声、风场、地图 seed
2
Collect rollouts in current training environment
策略在当前 Etr 中采样轨迹，填充 rollout buffer / replay buffer。这一步是训练的主采样阶段。
每 episode

episode 结束时形成摘要

  • success / crash / timeout
  • 累计违规次数、首次违规位置、最小安全距离
  • episode return、per-term return、task utility proxy
每 episode

更新可复用的统计缓存

把 episode summaries 累积成按约束、按环境 seed、按场景族统计的表，供后续 φ̄(1)EC、φ(2)CR 等读取。

3
完成一个 rollout window / batch buffer
当达到 horizon、episode 数或 sample budget 后，冻结这一批轨迹，形成一次可用于 policy update 与 CRE snapshot 的数据窗口。
说明

此时通常还不直接改 R/C/E

默认策略是：先累计证据，再判断是否达到 CRE gate。单个 episode 或单个 rollout window 通常不足以支撑结构性修改。

每 update

附加优化期日志

  • policy loss / value loss / entropy
  • critic fit quality、KL drift、gradient norms
  • 这些量给 M3 / discrepancy protocol 提供支持
4
执行优化：PPO/SAC update epochs / minibatches
用当前 rollout buffer 做若干轮 policy / critic 更新。这里是标准 RL 优化主段，通常不在 minibatch 中途插入 CRE 结构性 repair。
M3 候选输入

机制层信号在这里产生

如 reward-gradient drift、critic reliability、gradient conflict 等，不必每步都计算，但在 gate 时可从最近的训练窗口回读。

读取

CRE snapshot builder 取数

  • 当前 spec version
  • 最近 K 个 rollout windows 的轨迹与 episode summaries
  • 最近 checkpoint、critic 状态、eval logs
5
到达 CRE gate？（例如每 K 次 collect / 每 M 次 update / 每个 checkpoint）
这是 CRE 是否介入的关键闸门。建议在离散 gate 上运行 CRE，而不是在每个 step 或每个 minibatch 后立刻改规范。
每 gate

离线补充评估

  • 在 deployment-like envs 上做短评估
  • 收集 reward vs utility 的 cross-env 对齐信号
  • 必要时触发 utility oracle / semantic evaluator
读取日志

M2 / M3 读取哪些证据

  • M2：boundary clearance、coverage、activation、reward-utility scores
  • M3：gradient conflict、soft coverage similarity、gradient drift
6
运行 CRE diagnosis：M2 → M3 → DP(M6) → M5
输出 canonical reporters、enhanced reporters、discrepancy flags、ΨCRE、CI、failure modes。这里是训练中期诊断，不是训练完全结束后才做。
输出

产出的训练中期诊断结论

  • φ(2)CR, φ̄(1)EC, φ(3)ER
  • ΨCRE 与置信区间
  • LATENT_INCONSISTENCY / CRITIC_QUALITY_WARN
  • witness trajectories / failing scenes / active dims
Decision Gate：ΨCRE 是否低于 alarm threshold？
若 未触发告警：训练继续，进入下一轮 collect rollouts。
若 触发告警：进入 repair loop，但 repair 的提交仍然发生在“下一个训练窗口开始前”。
M4 / M7

语义诊断与 repair proposal

LLM 把数值诊断翻译为失败机制，并生成 proposal；不是每 episode 做，而是在告警 gate 上做。

7
若触发告警：M4 → M7(gen / filter / rank) ↔ M8(acceptance loop)
评估 proposals 是否同时满足 diagnostic improvement、safety non-regression、utility non-regression、minimality 与 semantic consistency。
可改什么

接受 repair 时允许修改的参数

  • R：wₖ、clip bounds、InjectPenalty、ModifyShaping
  • C：thresholds、severity、ζ、curriculum schedule
  • E：critical scene injection、domain rand range、spawn/obstacle mix
  • Eval：utility oracle config、Edep scene set、aggregation mode
写回时机

accepted repair 在哪里生效

通常在下一次 collect rollouts 之前：即 window 边界提交，而不是在当前 minibatch update 中途强行换规范。

8
写回 S′，重建或热更新训练部件，然后继续训练
根据 repair 类型执行热更新或冷更新：有些只需刷新 config；有些需要重建 env sampler、reward evaluator、constraint checker，甚至清理部分 buffer。
更新模式

典型生效方式

  • 热更新：env distribution range、scene mix、threshold、penalty weight
  • 部分重置：清空窗口统计、重建 rollout collector
  • 较重更新：reward graph / oracle 变化大时，从 checkpoint 恢复后继续
循环

继续记录“repair 之后”的新证据

repair 生效后，新的 step / episode 日志会形成 repair 前后的对照证据，用于验证是否真的改善。

9
返回训练主循环：继续 collect → update → gate → CRE
训练期 CRE 不是一次性后处理，而是伴随训练阶段反复运行的中期监控与修复闭环。
混合介入

真正的介入节奏

  • 每 step：采信号，不改规范
  • 每 episode：聚合摘要，不改主规范
  • 每 gate：诊断、打分、可能 repair
  • accepted repair：在下个 rollout window 前提交
训练结束

全程审计可追溯

保存每个 spec version、每次告警、每轮 proposal、每次 acceptance verdict，便于论文分析与复现实验。

10
训练结束后做 final CRE audit 与 deployment validation
最终输出：最佳 checkpoint、最终 spec version、CRE 演化轨迹、diagnostic history 与 deployment validation 结果。
post-train

训练后仍可再做一次完整 CRE

这不是唯一介入时机，而是对训练中多轮介入结果的最终审计与封板。

你最关心的三个问题，图里的答案

  • “是在每次 Collect rollouts 就更新环境或 constraint 吗？”
    不是默认行为。每次 collect rollouts 会更新日志；只有当达到 CRE gate 且告警成立、repair 被接受时，才会在后续训练窗口前更新。
  • “是在每个 episode 之后介入吗？”
    每个 episode 后会做 summary 与聚合，但通常不做结构性 R/C/E 改写。episode 级更多是证据累积层。
  • “是在 training 之后介入吗？”
    也不是只在训练后。标准做法是训练中期就周期性介入，训练后再做 final audit。因此是混合式：训练中周期诊断 + 训练后最终验证。

建议你在代码里对应成的触发粒度

  • Step hook：记录 reward terms、clearance、constraint indicators、env params。
  • Episode hook：写 episode summary、success/crash、violation aggregates。
  • Rollout-window hook：冻结最近 K 个 rollout windows，生成 CRE snapshot。
  • Checkpoint / eval hook：运行 M2/M3/DP/M5，必要时进入 M4/M7/M8。
  • Apply-repair hook：在下一轮 collect rollouts 之前更新 reward / constraint / env sampler / oracle config。

这会让“训练主干”和“CRE 外环”在工程上清楚分离：训练循环高频执行，CRE 诊断低频但更重。
